# 项目概述

这是一个名为"重生之我是福州古厝修复者"的网页游戏项目。横板RPG. 玩家扮演古建筑修复者，依次修复福州的三处历史文化街区：上下杭、朱紫坊、三坊七巷。每个章节包含战斗和修复两个阶段。

- 战斗阶段: 进入子页面完成小游戏. 失败主角死亡    
- 修复阶段: 连点古厝开始修复. 古厝依次会倒塌/爆炸(AI留炸弹), 出现主角死亡  

## 技术栈

- 后端：Django (Python)
- 前端：原生 HTML/CSS/JavaScript（单页应用）
- 数据存储：Session（无数据库）

## API 接口(localhost:10203)

### GET /api/progress/
获取当前游戏进度。

返回示例：
```json
{
    "chapter": 0,
    "chapter_name": "上下杭",
    "finished": false,
    "fight_won": false,
    "repaired": false,
    "relics": {
        "shangxiahang": false,
        "zhuzifang": false,
        "sanfangqixiang": false
    }
}
```

### POST /api/progress/advance/
推进游戏进度。

请求体：
```json
{ "action": "fight_win" | "repair_complete" | "reset" }
```

返回更新后的进度状态（格式同上）。

4. 任意时刻可 reset 重置

## 4. JavaScript 编码规范（强类型 + 现代写法）

### 4.1 类型与静态检查（核心要求）

* 每个 JS 文件顶部必须加：

  * `// @ts-check`
* 使用 **JSDoc** 写类型：

  * `@typedef` 定义对象结构
  * `@param` / `@returns` 约束函数签名
  * `@throws` 描述异常语义
* 所有导出/公共函数必须有 JSDoc；模块内关键私有函数也建议有

示例（你可以把它当作 JS 的“函数 docstring”标准）：

```js
// @ts-check

/**
 * 计算玩家修复进度百分比。
 * @param {number} doneCount 已完成数量
 * @param {number} totalCount 总数量
 * @returns {number} 百分比（0-100）
 * @throws {RangeError} 当 totalCount 小于等于 0
 */
function calcProgress(doneCount, totalCount) {
  if (totalCount <= 0) {
    throw new RangeError("totalCount 必须大于 0");
  }
  return Math.min(100, Math.max(0, (doneCount / totalCount) * 100));
}
```

### 4.2 语言特性与禁用项

* 必须使用：`const` / `let`，禁止 `var`
* 必须使用严格模式（若非 ES module）：

  * 内联脚本推荐 IIFE 包裹：`(() => { "use strict"; ... })();`
* 推荐使用：

  * 可选链：`obj?.a?.b`
  * 空值合并：`x ?? defaultValue`
  * 解构：`const { a, b } = obj`
  * 模板字符串：`` `${name}` ``
  * `Array.prototype.map/filter/reduce`
* 避免使用：

  * 隐式类型转换（例如 `==`），统一用 `===`
  * 过度嵌套回调（优先 async/await）
  * 随意的全局变量（必须封装在模块/IIFE）

### 4.3 错误处理与可观测性

* 允许失败的操作（网络/存储/解析）必须：

  * 明确 `try/catch`
  * 抛出**语义化错误**或返回明确错误对象（不要静默失败）
* 只在“边界层”做 `console.*`（例如调试入口），业务逻辑层不要散落日志

### 4.4 DOM 操作规范

* DOM 查询必须缓存：重复使用的节点不要重复 `querySelector`
* 事件绑定优先事件委托（特别是列表/动态节点）
* 严禁拼接不可信 HTML：不要 `innerHTML = userInput`
* UI 状态变化必须可追踪：推荐统一 `render(state)` 或最小化状态机

## 5. CSS 编码规范（与你“尽量简单”兼容）

### 5.1 命名与组织

* 推荐 BEM（适合小游戏/弹窗组件化）：

  * `dialog`（Block）
  * `dialog__header`（Element）
  * `dialog--open`（Modifier）

### 5.2 变量与主题

* 必须使用 CSS 变量管理主题与尺寸（像素风尤其需要统一尺度）：

  * `--ui-scale`
  * `--color-bg`
  * `--color-fg`
  * `--px`（像素风基准）
* 颜色、间距、层级（z-index）必须集中定义，禁止散落魔法数

### 5.3 布局与响应式

* 优先 Flex / Grid
* 移动端优先：使用 `@media (max-width: ...)`
* 触控区域：按钮最小点击区域建议 >= 44px（移动端）

### 5.4 禁止项

* 禁止 `!important`（除非第三方样式覆盖且不可控，并必须写明原因在文档字符串里）
* 禁止使用过深选择器链（>3 层通常就是维护风险）

风格规范:
## 1. 核心设计哲学 (Design DNA)

* **骨架 (Structure):** 档案馆式信息架构 + 8pt 网格系统。布局稳定、层级清晰、留白克制；强调“记录/条目/编号/刻度”的秩序感，而非现代营销式大块 Hero。
* **材质 (Materiality):** 纸张 / 木牌 / 石碑 三材质体系。质感来自微噪点、内描边、短阴影与轻微错位（手工痕迹），拒绝玻璃拟态与过度柔光阴影。
* **像素语言 (Pixel Language):** 像素只用于小规模符号化元素（图标、标记、角标、分隔、纹样），容器与排版保持现代可读。像素化是“统一风格的语法”，不是把整个 UI 画成像素画。
* **对立叙事 (Opposition):** “修复”侧温暖克制（纸木石）， “AI/结界”侧冷硬规则化（错误码、规则栅格、断续扫描线），但整体仍保持低饱和与可读性优先。

## 3. 排版与字体 (Typography)

* **字体：** 中文优先系统无衬线（可读、务实），数字/编号使用等宽字体。

  * UI 字体：`system-ui, -apple-system, "Segoe UI", "PingFang SC", "Microsoft YaHei", sans-serif`
  * 数字/编号：`"JetBrains Mono", ui-monospace, SFMono-Regular, Menlo, Consolas, monospace`
* **层级：** 标题 20–24；分区标题 16–18；正文 14–16（行高 1.6）；注释 12–13。
* **文案气质：** 档案条目/任务记录语气，短句直白；禁止营销腔与“灵感散文”。

---

## 4. 形状语言与细节 (Shape & Detail Language)

* **角与边框：** 二选一并全站统一

  1. 像素阶梯角（推荐），或
  2. 固定圆角（如 8px/12px），但不可混用。
* **描边：** 统一 1px 或 2px，强调“印刷/刻线”感；重要卡片可加 1px 内描边高光。
* **阴影：** 仅短阴影（类似纸张压在桌面），如 `0 2px 0 rgba(26,26,26,0.22)`；禁止扩散型大阴影与发光。
* **纹理（必须可解释）：**

  * 纸张：极淡纤维线 + 微噪点（透明度 < 6%）
  * 木牌：极淡木纹（规则、方向一致）
  * 石碑/石灰：颗粒感 + 细裂纹/补缝线（极淡）
    禁止不可解释的随机花纹与装饰线条堆砌。

---

## 5. 动态与交互 (Motion & Micro-interactions)

* **时长：** 120–180ms，`ease-out`；严禁超过 400ms。
* **Hover：** 轻微上移 2px 或内描边增强；按钮轻微变暗（brightness 95%）+ `scale(1.01)`。
* **Active：** 必须有按压反馈 `scale(0.98)` + 下沉 1–2px（短阴影减弱）。
* **Focus：** 明确可见（描边加粗或外圈细 ring），但风格保持“纸面标注”而非霓虹光晕。
* **动效原则：** 所有动效仅用于反馈与层级切换，拒绝弹跳、夸张缓动与炫技动效。

---

## 6. 图标与像素元素 (Iconography)

* **图标风格：** 仅允许像素网格图标（16/24/32），线性图标库一律禁用。
* **使用范围：** 角标、状态点、标记、按钮小图标、分隔符；不可把大面积插画当背景。
* **一致性：** 同一页面不得混用不同像素密度/不同描边粗细。

## 7. 禁止事项 (Strict Constraints)

1. 禁止玻璃拟态（backdrop blur）、霓虹渐变、赛博紫蓝大面积铺色。
2. 禁止 3D 插画、通用 SaaS 插画、Emoji、线性图标库混用。
3. 禁止柔和大阴影、强发光、高饱和大面积点缀。
4. 禁止不可解释的随机纹理与装饰线条堆叠。
5. 禁止营销模板化结构与“现代仪表盘模板”既视感；必须保持档案/记录的叙事气质。
6. 禁止同页出现超过 3 种字重；主要使用 Regular/Medium/Semibold。
